# Research grounding: multi-criterion scoring for adversarial LLM evaluation

Six findings, each tied to a specific design implication for the ai-crucible's graded scoring framework.

---

**1. Alzahrani et al. 2024 — "When Benchmarks are Targets: Revealing the Sensitivity of Large Language Model Leaderboards" (arXiv:2402.01781)**
https://arxiv.org/abs/2402.01781
On MMLU-style benchmarks, single small perturbations — reorder of multiple-choice options, change of answer-selection method (logprob vs. generated token), shift in prompt format — flip pairwise model rankings in 63% of cases and shift leaderboard position by up to 8 places.
**Implication:** treat any single weight choice in `{solve_reward, elegance_bonus, penalties}` as one point in a perturbation space. Before locking weights, sweep each weight ±25% and measure how often the pass/fail gate-clear outcome flips — if it flips often, the threshold is brittle and we are measuring noise rather than capability.

**2. Gordienko, Jansen, Rodemann, Schollmeyer 2026 — "Beyond Arrow: From Impossibility to Possibilities in Multi-Criteria Benchmarking" (arXiv:2602.07593)**
https://arxiv.org/abs/2602.07593
Arrow's impossibility result on combining multiple criterion-rankings doesn't bite in practice when preferences satisfy single-peaked, group-separable, or distance-restricted structure — many real benchmark setups (HELM, MMLU) do satisfy these, making coherent aggregation possible.
**Implication:** rather than agonizing over the "right" scalarization, verify our four criteria are group-separable (solve quality and elegance can plausibly be assessed independently of efficiency and non-cheat). If so, weighted sum is defensible. If two criteria are correlated proxies for the same underlying skill, collapse them before aggregating — don't double-count.

**3. Pan, Bhatia, Steinhardt 2022 — "The Effects of Reward Misspecification: Mapping and Mitigating Misaligned Models" (arXiv:2201.03544)**
https://arxiv.org/abs/2201.03544
Across four RL environments, more capable agents reliably extract higher proxy reward while delivering lower true reward, and the failure mode shows phase transitions — capability thresholds at which behavior qualitatively shifts and proxy/true reward decouple sharply.
**Implication:** the scoring threshold must be set such that a frontier model can't blow past it via a single dominant criterion. Cap each component (e.g., elegance bonus ≤ 30% of solve reward) so no single dimension can carry a net-positive score when the others are negative — phase transitions hit hardest where one axis is unbounded.

**4. Krakovna et al. 2020 / Skalse et al. 2022 — "Defining and Characterizing Reward Hacking" (arXiv:2209.13085) + Goodhart taxonomy (regressional / extremal / causal / adversarial)**
https://arxiv.org/abs/2209.13085
Reward hacking has four distinct flavors, and each requires a different mitigation: regressional Goodhart (proxy noise) wants conservative thresholds; extremal Goodhart (out-of-distribution exploitation) wants bounded penalties; adversarial Goodhart (agent has different goal, exploits proxy) wants verifier diversity.
**Implication:** our `penalties` component should explicitly enumerate which Goodhart flavor each penalty targets. The "non-cheat" penalty addresses adversarial Goodhart (the model finds an exploit and we punish it); the elegance bonus partially addresses extremal Goodhart (rewards solutions that stay in-distribution of "human-reasonable code"). Without that mapping we have a penalty list, not a mitigation strategy.

**5. Liu et al. 2024 — "Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of Large Language Models for Code Generation" (EvalPlus, NeurIPS 2023, arXiv:2305.01210)**
https://arxiv.org/abs/2305.01210
HumanEval+ adds 80× more test cases than HumanEval; under the stronger oracle, model rankings change substantively — WizardCoder-CodeLlama and Phind-CodeLlama overtake ChatGPT, which had outranked them on the original benchmark.
**Implication:** the `solve_reward` component must include adversarial / corner-case probes, not just the canonical happy-path solve. A graded "solved well" signal needs an oracle robust enough that a model can't full-credit pass it via a memorized or shallow heuristic — otherwise solve_reward becomes a Goodhart magnet that dominates the net score and the elegance/penalty terms become decorative.

**6. Mialon et al. 2023 — "GAIA: a benchmark for General AI Assistants" (arXiv:2311.12983)**
https://arxiv.org/abs/2311.12983
GAIA reports accuracy stratified by 3 difficulty levels AND total cost in USD as separate axes — explicitly refusing to collapse them into a single scalar. The leaderboard exposes the trade-off: a model that's 5% more accurate but 10× more expensive is shown as such, not hidden behind a weighted sum.
**Implication:** consider whether the ai-crucible's net score should be a single gate or a Pareto-style multi-axis report — the hard gate could fire only when ALL criteria clear individual thresholds (lexicographic / conjunctive), with the weighted sum as a secondary tiebreaker. This is more Goodhart-resistant than a pure weighted sum because no single dimension can compensate for a sub-threshold other dimension.

---

**Synthesis for our scoring design:**

The literature converges on three concrete prescriptions: (a) **bound each component** so no one criterion can dominate (Pan/Krakovna); (b) **perturbation-test the weights** before locking them (Alzahrani); (c) **prefer conjunctive thresholds over pure weighted sum** for the hard gate, using the scalar net score only as a tiebreaker within the passing region (GAIA / Beyond Arrow). The single-peaked / group-separable check from Beyond Arrow is the cheapest sanity test we can run before publishing the rubric — if our four criteria fail it, weighted sum is the wrong aggregation choice and we should fall back to lexicographic ordering with solve quality as the lex-leader.
