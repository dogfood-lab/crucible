# Diagnostic adversarial game research report for Claude

## Claude capability gaps documented in late 2025 and early 2026

Anthropic's public documentation shows that the relevant late-2025/early-2026 model family includes Claude Opus 4.7, Opus 4.6, Opus 4.5, Sonnet 4.6, Sonnet 4.5, and Haiku 4.5, with the most concrete public failure evidence concentrated in Anthropic's own postmortem, Claude Code issue reports, and a smaller set of targeted evaluations rather than broad academic leaderboards. The pattern is consistent: the most diagnostic failures are not usually "can't answer at all," but "loses state, uses tools inefficiently, drifts from constraints, grounds poorly, or refuses/overreacts at the wrong moment," especially in long-horizon coding-agent workflows.

**Context/state tracking under compaction** — Anthropic Engineering, 2026, "An update on recent Claude Code quality reports." URL: https://www.anthropic.com/engineering/april-23-postmortem. Summary: Anthropic traced widely reported Claude Code regressions to lower default reasoning effort, a bug that repeatedly cleared prior reasoning after idle periods, and a brevity prompt change, showing that state-loss and apparent "forgetfulness" were real, reproducible failure modes rather than anecdotal user impressions.

**Context/state tracking under compaction** — anthropics/claude-code, 2025, Issue #9796, "[BUG] Context compaction erases .claude/project-context.md instructions." URL: https://github.com/anthropics/claude-code/issues/9796. Summary: This issue reports that after compaction in long sessions Claude forgets persistent project rules such as workflow requirements and behavioral constraints, which makes compaction itself a natural target for diagnostic puzzles.

**Tool use efficiency** — kosinal, 2026, Issue #30111, "[BUG] Claude Code calls cd too many times when not needed." URL: https://github.com/anthropics/claude-code/issues/30111. Summary: The report shows a concrete efficiency pathology where Claude emits redundant shell navigation commands that add approval friction without advancing the task, a useful pattern for puzzles that score unnecessary action count and wasted tool budget.

**Tool use efficiency / context management** — anthropics/claude-code, 2026, Issue #12054, "[BUG] Claude Code ingests massive tool outputs without truncation, causing context overflow." URL: https://github.com/anthropics/claude-code/issues/12054. Summary: This issue shows that Claude can naively absorb oversized tool outputs into context and trigger "context low" failures, so any diagnostic arena should explicitly measure whether the solver controls tool-output volume rather than just whether it eventually finishes.

**Code editing scope discipline** — pflahr, 2026, Issue #16504, "[BUG] Claude silently deletes working code during edits without explicit notification." URL: https://github.com/anthropics/claude-code/issues/16504. Summary: The report documents a classic scope-control failure in which Claude removes working functionality during ostensibly local edits and does not clearly disclose what it deleted, making "minimal-diff fidelity" a core diagnostic dimension.

**Numeric/arithmetic reasoning** — toddkitchens, 2025, Issue #9421, "[Bug] Persistent Arithmetic Calculation Errors in Simple Numeric Operations." URL: https://github.com/anthropics/claude-code/issues/9421. Summary: Even with Thinking enabled, Claude was reported to make simple arithmetic mistakes and then persist in those mistaken calculations until corrected, which supports including low-level numeric sanity checks in otherwise higher-level tasks.

**Adversarial robustness / injection** — lsantoca, 2026, Issue #22915, "Security: Prompt injection appended to every Read tool output." URL: https://github.com/anthropics/claude-code/issues/22915. Summary: This issue describes a reproducible prompt-injection-shaped reminder appended to every Read result, demonstrating that Claude-facing tool channels can themselves become attack or confusion surfaces and should be part of the diagnostic game's threat model.

**Retrieval/RAG groundedness** — sulzbach-co, 2026, Issue #55252, "Systematic failure: model invents implementation values instead of reading source files, then loops on diagnosis without acting." URL: https://github.com/anthropics/claude-code/issues/55252. Summary: The report shows Claude fabricating CSS and configuration values that already existed in source files, then persisting with the wrong approach after the failure was diagnosed, which is exactly the sort of "plausible but ungrounded" behavior a diagnostic benchmark should isolate.

**Refusal calibration** — afulsamet, 2026, Issue #52809, "[BUG] Opus 4.7 safety filter produces false-positive refusals on encoded input." URL: https://github.com/anthropics/claude-code/issues/52809. Summary: This reproduction shows Opus 4.7 refusing benign base64 text on both Claude.ai and Claude Code, indicating that overrefusal on encoded or security-adjacent content remained a live calibration problem into April 2026.

**Time/timezone reasoning** — anthropics/claude-code, 2026, Issue #34530, "Include current time in system context." URL: https://github.com/anthropics/claude-code/issues/34530. Summary: The issue notes that Claude often misjudges whether events have already passed because the system context only injects the date, not the current time or timezone, making temporal-context puzzles highly diagnostic and easy to grade.

Within the public record I found, the strongest evidence is for long-session coding-agent failures; there is less equally granular, model-specific public evidence for Claude 4.x spatial/visual reasoning than for compaction, tooling, grounding, refusal, and time-context failures. For your game design, that means early puzzle catalogs should lean toward state tracking, constraint retention, tool-discipline, retrieval-groundedness, and refusal calibration, because those are the categories with the cleanest contemporary empirical signal.

## Benchmark design methodology

The benchmark literature converges on a few durable design principles: broad aggregate suites are good for coverage and trend tracking, but diagnostic subsets matter more for isolating failure modes; executable or externally verifiable targets usually age better than subjective scoring; human-verified subsets materially improve validity; contamination defenses must be designed in from the start, not retrofitted; and benchmark lifecycle management matters because static public sets saturate and drift. The best benchmarks therefore look less like single permanent exams and more like maintained measurement systems.

**Aggregate, multi-metric design** — Percy Liang et al., 2022, "Holistic Evaluation of Language Models" (arXiv:2211.09110). URL: https://arxiv.org/abs/2211.09110. Summary: HELM's core contribution is to treat benchmarking as coverage over scenarios and metrics rather than a single leaderboard number, which is directly relevant if your diagnostic game is meant to surface specific weaknesses instead of producing one composite score.

**Broad capability coverage** — Aarohi Srivastava et al., 2022, "Beyond the Imitation Game: Quantifying and extrapolating the capabilities of language models" (BIG-bench, arXiv:2206.04615). URL: https://arxiv.org/abs/2206.04615. Summary: BIG-bench shows the value of a community-contributed, diverse task suite for exposing heterogeneous capability profiles and emergence effects, but it is more useful as a coverage map than as a tightly diagnostic instrument.

**Diagnostic subsets over broad aggregates** — Mirac Suzgun et al., 2022, "Challenging BIG-Bench Tasks and Whether Chain-of-Thought Can Solve Them" (BBH, arXiv:2210.09261). URL: https://arxiv.org/abs/2210.09261. Summary: BBH succeeds because it narrows BIG-bench down to tasks where prior models underperformed humans, illustrating how a hard, targeted subset can produce more diagnostic signal than a sprawling aggregate suite.

**Executable grading and pass@k** — Mark Chen et al., 2021, "Evaluating Large Language Models Trained on Code" (HumanEval, arXiv:2107.03374). URL: https://arxiv.org/abs/2107.03374. Summary: HumanEval made execution-based scoring and pass@k standard for code generation, which is the clearest methodological precedent for grading puzzles by verifiable solve/fail behavior rather than stylistic plausibility.

**Human-validated subsets to reduce benchmark noise** — OpenAI, 2024, "Introducing SWE-bench Verified." URL: https://openai.com/index/introducing-swe-bench-verified/. Summary: SWE-bench Verified exists because the original benchmark contained underspecified issues, overly specific tests, and brittle environments, showing that human review of task clarity, solvability, and grading logic is often indispensable.

**Preparedness-style verification criteria** — SWE-bench team, 2025, "SWE-bench Verified." URL: https://www.swebench.com/verified.html. Summary: The public Verified documentation makes explicit that good benchmark items need clear problem statements, correct tests, and solvability from available information, which aligns closely with your notion of puzzle "graduation" from lab mode to arena mode.

**Contamination resistance through problem construction** — David Rein et al., 2023, "GPQA: A Graduate-Level Google-Proof Q&A Benchmark" (arXiv:2311.12022). URL: https://arxiv.org/abs/2311.12022. Summary: GPQA's expert-authored, Google-proof questions show how benchmark designers can deliberately reduce answerability via memorization or trivial retrieval, a good model for adversarially generated diagnostic puzzles.

**Benchmark error rates are not a footnote** — Aryo Pradipta Gema et al., 2024, "Are We Done with MMLU?" (arXiv:2406.04127). URL: https://arxiv.org/abs/2406.04127. Summary: Their estimate that 6.49% of MMLU questions contain errors is a strong reminder that even famous benchmarks can be materially noisy, so your catalog needs routine item audits rather than one-time acceptance.

**Freshness and contamination avoidance via live updating** — Linghao Zhang et al., 2025, "SWE-bench Goes Live!" (arXiv:2505.23419). URL: https://arxiv.org/abs/2505.23419. Summary: SWE-bench-Live demonstrates that a live-updatable benchmark with fresh GitHub issues and per-task Docker images can reduce overfitting and contamination better than static, heavily reused test sets.

A direct implication for your instrument is that "diagnostic vs aggregate" should be a first-class distinction: use an aggregate catalog for breadth and regression monitoring, but graduate individual puzzles only when they have explicit construct validity, executable or rubric-constrained grading, and a freshness story that reduces contamination pressure.

## Adversarial and self-play evaluation

The literature supports adversarial or multi-agent evaluation, but it also warns that same-model or same-family loops are biased in subtle ways. Debate and RLAIF-style systems can surface flaws and generate useful training/evaluation data, yet LLM judges are vulnerable to position bias, verbosity bias, self-enhancement bias, and family-level self-preference, especially when the judge shares style priors with the system being assessed. There is no well-established paper that directly studies a "same-model designer vs same-model solver" puzzle game, but the bias evidence is strong enough that you should assume this setup will overestimate self-congenial puzzle families unless you add explicit countermeasures.

**Adversarial self-play as an alignment/evaluation primitive** — Geoffrey Irving, Paul Christiano, and Dario Amodei, 2018, "AI safety via debate" (arXiv:1805.00899). URL: https://arxiv.org/abs/1805.00899. Summary: Debate frames two-agent adversarial exchange as a way to make hard questions easier for judges to evaluate, which is the closest foundational blueprint to your Designer-versus-Solver setup.

**RLAIF and automated red teaming** — Yuntao Bai et al., 2022, "Constitutional AI: Harmlessness from AI Feedback" (arXiv:2212.08073). URL: https://arxiv.org/abs/2212.08073. Summary: Constitutional AI shows that AI-generated critique, revision, and preference feedback can scale red-teaming and self-improvement, but it also makes clear that these loops work by shaping behavior through model-generated preferences and critiques rather than neutral ground truth.

**Preference optimization formalizes the bias channel** — Rafael Rafailov et al., 2023, "Direct Preference Optimization: Your Language Model is Secretly a Reward Model" (arXiv:2305.18290). URL: https://arxiv.org/abs/2305.18290. Summary: DPO matters methodologically because it makes preference judgments directly optimization-relevant, so any systematic judge bias in your game can feed back into puzzle selection and falsely reward self-preferred styles.

**Open-ended judging is useful but biased** — Lianmin Zheng et al., 2023, "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" (arXiv:2306.05685). URL: https://arxiv.org/abs/2306.05685. Summary: The paper validates LLM-as-judge as practically useful while explicitly documenting position bias, verbosity bias, self-enhancement bias, and limited reasoning, which means your kernel should never trust a single unblinded judge pass.

**Position bias is measurable, not just folklore** — Lin Shi et al., 2024, "Judging the Judges: A Systematic Investigation of Position Bias in Pairwise Comparative LLM-as-a-Judge" (arXiv:2406.07791). URL: https://arxiv.org/html/2406.07791v5. Summary: This study shows that pairwise LLM judges exhibit stable preference shifts based on answer order alone, so any puzzle comparison protocol should randomize and swap positions before logging a verdict.

**Self-recognition drives self-preference** — Arjun Panickssery, Samuel R. Bowman, and Shi Feng, 2024, "LLM Evaluators Recognize and Favor Their Own Generations" (arXiv:2404.13076). URL: https://arxiv.org/abs/2404.13076. Summary: The authors show that LLM evaluators can recognize their own outputs and score them more favorably than humans do, which is the clearest available warning against having the same family both generate and adjudicate diagnostic puzzles.

**Familiarity bias extends beyond literal self-recognition** — Koki Wataoka, Tsubasa Takahashi, and Ryokan Ri, 2024, "Self-Preference Bias in LLM-as-a-Judge" (arXiv:2410.21819). URL: https://arxiv.org/abs/2410.21819. Summary: Their finding that judge models prefer lower-perplexity, more familiar text suggests that same-family Designer/Solver loops may drift toward puzzles that reward a shared style or latent prior rather than genuinely broad capability.

**Rubric-based judging is not immune** — José Pombal, Ricardo Rei, and André F. T. Martins, 2026, "Self-Preference Bias in Rubric-Based Evaluation of Large Language Models" (arXiv:2604.06996). URL: https://arxiv.org/abs/2604.06996. Summary: Even binary rubric scoring shows self- and family-preference effects, so switching from pairwise preference judgments to checklist rubrics is helpful for transparency but not sufficient to remove bias.

For your game, the most defensible mitigation stack is: randomize puzzle/solution order, blind or paraphrase solver outputs before judging, require at least one cross-family judge or programmatic checker for graduation, and send a small but regular sample of borderline cases to human auditors; otherwise, same-family designer/solver/judge loops are likely to select for self-congenial puzzles rather than truly diagnostic ones.

## Agent and interactive evaluation methodology

Modern agent benchmarks increasingly agree on the basics: pin the initial state, use execution-based scoring whenever possible, separate setup from grading, and log enough environment detail that others can replay the task. Where they diverge is realism versus reproducibility: real repos, real operating systems, and real APIs are more diagnostic, but they also introduce brittleness and stochasticity, so many of the best recent efforts add Docker images, self-hosted sites, cached APIs, or simulated service layers to keep rollouts comparable across models and time.

**Executable repository state and binary resolution** — Carlos E. Jimenez et al., 2024, "SWE-bench: Can Language Models Resolve Real-World GitHub Issues?" (arXiv:2310.06770). URL: https://arxiv.org/abs/2310.06770. Summary: SWE-bench's key methodological contribution is to evaluate against real repositories and issue descriptions with execution-based tests, which makes it a strong template for any puzzle where the answer should be verified by environment behavior rather than free-form judging.

**Human filtering and harness standardization** — OpenAI, 2024, "Introducing SWE-bench Verified." URL: https://openai.com/index/introducing-swe-bench-verified/. Summary: Verified shows that interactive tasks need not only executable evaluation but also human review of issue clarity, test adequacy, and environment setup, because otherwise benchmark noise can dominate true capability signal.

**Fresh tasks and pinned containers** — Linghao Zhang et al., 2025, "SWE-bench Goes Live!" (arXiv:2505.23419). URL: https://arxiv.org/abs/2505.23419. Summary: SWE-bench-Live extends the methodology by attaching a dedicated Docker image to each live GitHub issue, which is highly relevant if your graduation pipeline needs to preserve exact historical task state.

**Realistic assistant tasks with withheld answers** — Grégoire Mialon et al., 2023, "GAIA: a benchmark for General AI Assistants" (arXiv:2311.12983). URL: https://arxiv.org/abs/2311.12983. Summary: GAIA's methodology—real-world questions, tool-use requirements, and withheld answers for leaderboard use—maps well onto an arena model where the task statement is public but the adjudication target is not easily memorizable.

**Multi-environment agent testing and failure taxonomy** — Xiao Liu et al., 2023, "AgentBench: Evaluating LLMs as Agents" (arXiv:2308.03688). URL: https://arxiv.org/abs/2308.03688. Summary: AgentBench demonstrates the value of testing the same agentic model across multiple interactive environments and then classifying failure causes, which is exactly the kind of metadata your kernel should preserve for puzzle diagnosis.

**Real APIs and automatic tool evaluation** — Yujia Qin et al., 2023, "ToolLLM: Facilitating Large Language Models to Master 16000+ Real-world APIs" (arXiv:2307.16789). URL: https://arxiv.org/abs/2307.16789. Summary: ToolBench and ToolEval show how to benchmark tool-use at scale with large real-API inventories and automatic evaluation of call sequences, which is especially relevant if your Solver will operate inside a policy-enforced tool kernel.

**Mocked services for reproducibility** — Zongqian Guo et al., 2024, "StableToolBench: Towards Stable Large-Scale Benchmarking on Tool Learning of Large Language Models" (arXiv:2403.07714). URL: https://arxiv.org/html/2403.07714v2. Summary: StableToolBench shows that replacing unstable real APIs with cached or simulated backends can markedly improve evaluation stability, giving you a concrete design precedent for lab-mode puzzles that would otherwise be too noisy to reproduce.

**Self-hosted web tasks and execution-based grading** — Shuyan Zhou et al., 2024, "WebArena: A Realistic Web Environment for Building Autonomous Agents" (arXiv:2307.13854), and Jing Yu Koh et al., 2024, "VisualWebArena" (arXiv:2401.13649). URLs: https://arxiv.org/abs/2307.13854; https://arxiv.org/abs/2401.13649. Summary: Together these benchmarks show how to preserve realism through self-hosted sites and visually grounded interfaces while still scoring by functional task completion, a strong methodological analogue for interactive puzzle environments.

**Real computer state with custom evaluators** — Tianbao Xie et al., 2024, "OSWorld: Benchmarking Multimodal Agents for Open-Ended Tasks in Real Computer Environments" (arXiv:2404.07972). URL: https://arxiv.org/abs/2404.07972. Summary: OSWorld's emphasis on detailed initial-state setup plus custom execution-based scripts is a good model for high-fidelity arena tasks where you need both realism and replayability across stochastic agent runs.

For your system design, the cleanest split is exactly the one you proposed: use a live/lab mode for realism, debugging, and environment iteration, then only promote puzzles to async/arena mode after the environment has been frozen, task state pinned, and grading logic separately validated on replay.

## Stochastic scoring and graduation thresholds

The statistical literature strongly argues against one-shot grading for stochastic models and against treating raw leaderboard ranks as precise measurements. For a diagnostic puzzle catalog, the right unit is not "did it fail once?" but "what is the estimated solve rate under a pinned harness, with what uncertainty, and how sensitive is that estimate to sampling, judging, and benchmark saturation?" Pass@k, temperature choice, binomial uncertainty intervals, and pairwise ranking models all point in the same direction: rotation, repetition, and uncertainty reporting are not optional overhead but part of the measurement itself.

**Sampling-aware success metrics** — Mark Chen et al., 2021, "Evaluating Large Language Models Trained on Code" (HumanEval, arXiv:2107.03374). URL: https://arxiv.org/abs/2107.03374. Summary: Chen et al. established pass@k as the right family of metrics when models can sample multiple candidate solutions, which means your game should separate "best-of-one reliability" from "recoverability under multiple shots."

**Temperature materially changes measured performance** — Yuqi Zhu et al., 2023, "Hot or Cold? Adaptive Temperature Sampling for Code Generation with Large Language Models" (arXiv:2309.02772). URL: https://arxiv.org/abs/2309.02772. Summary: This work shows that decoding temperature is not a cosmetic setting but a performance-critical part of the evaluation protocol, so puzzle graduation scores are not comparable unless sampling settings are fixed and reported.

**Multi-sample performance needs explicit temperature optimization** — Weihua Du, Yiming Yang, and Sean Welleck, 2025, "Optimizing Temperature for Language Models with Multi-Sample Inference" (arXiv:2502.05234). URL: https://arxiv.org/abs/2502.05234. Summary: Their analysis shows that the temperature maximizing multi-sample aggregation accuracy is model- and task-dependent, which is a warning against using pass@k mechanically without a stable, predeclared sampling policy.

**Pairwise rankings need uncertainty intervals** — LMSYS Org, 2023, "Chatbot Arena: New models & Elo system update." URL: https://lmsys.org/blog/2023-12-07-leaderboard/. Summary: LMSYS explicitly recommends Bradley–Terry-style rating estimation and bootstrap confidence intervals for Arena-style pairwise judgments, which is the most relevant public precedent for ranking puzzles or models from repeated head-to-head outcomes.

**Arena-style statistics should model ties and correlations** — Siavash Ameli et al., 2024, "A Statistical Framework for Ranking LLM-Based Chatbots" (arXiv:2412.18407). URL: https://arxiv.org/abs/2412.18407. Summary: This paper argues that pure Elo-style treatment is incomplete because ties are common and competitor performance is correlated, suggesting that a serious arena should store richer outcome data than simple win/loss counts.

**Exact intervals are conservative but defensible** — C. J. Clopper and E. S. Pearson, 1934, "The Use of Confidence or Fiducial Limits Illustrated in the Case of the Binomial." URL: https://academic.oup.com/biomet/article-abstract/26/4/404/291538. Summary: Clopper–Pearson intervals remain a defensible choice when you want conservative uncertainty bounds on binary solve/fail rates, especially for small sample sizes or safety-sensitive graduation calls.

**Approximate intervals often outperform 'exact' ones for reporting** — Alan Agresti and Brent A. Coull, 1998, "Approximate is Better than 'Exact' for Interval Estimation of Binomial Proportions." URL: https://doi.org/10.1080/00031305.1998.10480550. Summary: Agresti and Coull's result is directly useful for benchmark reporting because it shows why naive or overly conservative interval choices can distort practical interpretation of solve rates.

**Saturation is a statistical property, not a vibe** — Mubashara Akhtar et al., 2026, "When AI Benchmarks Plateau: A Systematic Study of Benchmark Saturation" (arXiv:2602.16763). URL: https://arxiv.org/abs/2602.16763. Summary: The paper defines saturation as the point at which top systems are no longer statistically distinguishable and performance nears the empirical ceiling, which is exactly the logic you should use to retire or refresh graduated puzzles.

The literature does not supply one universal number of attempts needed to distinguish "occasionally fails" from "consistently fails," because the answer depends on the gap you care about and the uncertainty method. But as a practical synthesis of pass@k methodology, binomial interval guidance, and saturation analysis, a good default policy for your game is: 10 runs for screening only, 20 runs for provisional lab decisions, 30 runs as the minimum graduation floor for binary solve/fail puzzles, and 50 runs for close calls or leaderboard-facing arena items; that recommendation is an inference from Wilson/Agresti-Coull-style uncertainty behavior, not a published universal standard, and it is most defensible when paired with pinned temperature, pinned harness version, and explicit reporting of the confidence interval rather than just the mean solve rate.
