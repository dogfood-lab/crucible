# Swarm Agent 04 — Benchmark Designs That Surface Novel Gaps

**Date:** 2026-05-27
**Source:** Study swarm dispatch (research-grounded advisor protocol)
**Question:** What benchmark design patterns have proven effective at surfacing previously-unknown capability gaps, rather than just measuring known ones?

---

# Benchmark Design Patterns That Surface Novel Capability Gaps

**1. Chollet (2019) — "On the Measure of Intelligence" (arXiv:1911.01547)**
URL: https://arxiv.org/abs/1911.01547
Summary: ARC-AGI deliberately constrains tasks to a fixed set of "innate" priors (objectness, counting, basic geometry) and guarantees every task is unique, so scale-and-memorize approaches cannot buy performance.
**Design principle:** *Hold the prior knowledge envelope fixed and force every item to be unseen by construction — this converts the benchmark from a skill test into a skill-acquisition-efficiency test.*
Implication: Our catalog should specify the "allowed priors" explicitly and generate items that are guaranteed-novel along at least one axis, so improved memorization cannot quietly raise scores.

**2. Kiela et al. (2021) — "Dynabench: Rethinking Benchmarking in NLP" (arXiv:2104.14337)**
URL: https://arxiv.org/abs/2104.14337
Summary: Dynabench is a platform where annotators interactively craft examples a target model misclassifies but another human gets right, producing a moving benchmark indexed to the current frontier.
**Design principle:** *Make the benchmark a process, not a dataset — humans iteratively probe the current frontier for examples it fails on, which by construction tracks unknown gaps rather than known ones.*
Implication: We should treat the catalog as a continuously-growing artifact maintained by adversarial human probing against the current SOTA, not a frozen v1.0 list.

**3. Nie et al. (2019) — "Adversarial NLI: A New Benchmark for Natural Language Understanding" (arXiv:1910.14599)**
URL: https://arxiv.org/abs/1910.14599
Summary: ANLI was collected in three sequential rounds where each round's training data was the previous round's model, with non-expert annotators reliably finding new failure modes each round.
**Design principle:** *Rounds-against-rounds — retrain on round N, collect round N+1 against the retrained model, and the gaps you find in round N+1 are necessarily new ones.*
Implication: Our diagnostic should have explicit "rounds" tied to model generations, where each round's seed pool is the prior round's resolved failures, guaranteeing the catalog stays ahead of the current frontier.

**4. Mialon et al. (2023) — "GAIA: a benchmark for General AI Assistants" (arXiv:2311.12983)**
URL: https://arxiv.org/abs/2311.12983
Summary: GAIA targets questions humans find trivial (92%) but frontier models fail (15%), inverting the usual difficulty axis to expose composition-and-tool-use gaps that single-skill benchmarks miss.
**Design principle:** *Invert the difficulty axis — seek items that are easy for humans but hard for models, since these mark genuine capability gaps rather than mere computational difficulty.*
Implication: Our gap-detection items should be filtered through a human-easy / model-hard sieve; any item that is also hard for humans is measuring something else.

**5. Suzgun et al. (2022) — "Challenging BIG-Bench Tasks and Whether Chain-of-Thought Can Solve Them" (arXiv:2210.09261)**
URL: https://arxiv.org/abs/2210.09261
Summary: BBH was constructed by filtering 200+ BIG-Bench tasks down to the 23 where contemporary LMs still underperformed humans, concentrating evaluation signal where gaps actually live.
**Design principle:** *Live-prune the catalog against the current frontier — keep only items where models still fail, so the benchmark's distribution stays where the gaps are.*
Implication: We should periodically demote items the frontier has solved into a "solved" archive and reweight the active catalog toward unsolved items, so headline scores remain informative.

**6. Liu et al. (2023) — "Is Your Code Generated by ChatGPT Really Correct?" / EvalPlus (arXiv:2305.01210)**
URL: https://arxiv.org/abs/2305.01210
Summary: HumanEval+/MBPP+ apply LLM-driven plus type-mutation test generation to expand test suites 35-80x, exposing failures that the original sparse tests hid (pass@k dropped 19-29%).
**Design principle:** *Stress-test the test set itself — sparse evaluation criteria mask capability gaps, so adversarially expanded oracles reveal failures the original benchmark misses.*
Implication: For every diagnostic item we should generate a multiplied oracle (mutated checks, edge cases, type-driven negatives) so "passing" cannot be achieved by shallow pattern-matching to the canonical test.

**7. Jain et al. (2024) — "LiveCodeBench" (arXiv:2403.07974)**
URL: https://arxiv.org/abs/2403.07974
Summary: LiveCodeBench continuously scrapes dated contest problems from LeetCode/AtCoder/CodeForces and lets evaluators filter to problems posted after a model's training cutoff, killing contamination.
**Design principle:** *Date every item and filter against model training cutoffs — provenance with timestamps is the only contamination-proof guarantee, far stronger than holdout splits.*
Implication: Every catalog item must carry a creation timestamp and source-origin metadata; evaluation runs should auto-restrict to post-cutoff items per model.

**8. Jimenez et al. (2024) / OpenAI (2024) — "SWE-bench" + "SWE-bench Verified" (arXiv:2310.06770)**
URL: https://arxiv.org/abs/2310.06770 (Verified report: https://openai.com/index/introducing-swe-bench-verified/)
Summary: SWE-bench draws tasks from real merged GitHub PRs across 12 repos, then SWE-bench Verified had 93 engineers filter out underspecified, broken-test, or environment-flaky instances, leaving 500 high-trust items.
**Design principle:** *Real-world provenance plus human filtering — sample from messy production artifacts to get genuine novelty, then human-verify to remove noise that would obscure capability signal.*
Implication: Our catalog should pull seeds from real artifacts (issues, tickets, transcripts) rather than synthetic constructions, but every accepted item must pass an explicit "solvable, well-specified, oracle-correct" gate.

**9. McKenzie et al. (2023) — "Inverse Scaling: When Bigger Isn't Better" (arXiv:2306.09479)**
URL: https://arxiv.org/abs/2306.09479
Summary: A public prize crowdsourced 80+ candidate tasks and selected 11 where larger models do worse, exposing failure modes (memorized-prior preference, distractor-task following) that simple scaling cannot fix.
**Design principle:** *Hunt for inverse-scaling items, not just hard items — tasks where bigger models perform worse reveal structural training-objective flaws that next-generation scaling will not paper over.*
Implication: Our diagnostic should explicitly maintain an "inverse-scaling" lane where item difficulty is calibrated against model size, surfacing the gaps that scaling actively makes worse.
