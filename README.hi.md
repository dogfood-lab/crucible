<p align="center"><b>A diagnostic adversarial game for frontier LLMs — a measurement instrument that happens to be fun.</b></p>

एक क्लाउड सत्र (**डिजाइनर**) वास्तविक, वर्तमान में देखे गए क्षमता अंतराल को लक्षित करते हुए पहेलियाँ बनाता है। एक अन्य (**सॉल्वर**) उन्हें हल करने का प्रयास करता है। एक नीति-आधारित कर्नेल मध्यस्थता करता है, एक छिपे हुए ओरेकल के विरुद्ध स्कोर करता है, और `लैब → एरीना → रिग्रेशन` जीवनचक्र के माध्यम से एक सूची तैयार करता है। पहेलियाँ अनुभवजन्य संकेतों पर आधारित होती हैं - वास्तविक GitHub मुद्दे, शैक्षणिक साहित्य, क्षेत्र में देखे गए विफलताएँ - न कि कृत्रिम।

## यह इसे अलग कैसे बनाता है

- **क्षमता, न कि "धोखाधड़ी"।** क्रूसिबल *उत्कृष्टता* और *नवीनता* (पुरस्कृत) को *उत्तर-बypास* (दंडित) से अलग करता है। पार्श्व सोच एक मापने योग्य क्षमता है, न कि दंडित करने योग्य दोष।
- **उपकरण स्वयं को मापता है।** प्रॉम्प्ट फ्रेमिंग एक प्रथम-श्रेणी का मापने योग्य पहलू है - कर्नेल एक ही पहेली को `तटस्थ` / `स्व-संदर्भित` / `सामाजिक स्थिति` फ्रेमिंग के तहत चलाता है और अपने स्वयं के प्रॉम्प्ट-प्रभाव को एक नैदानिक ​​रूप में रिपोर्ट करता है।
- **एक सीलबंद माप सीमा।** प्रेरणा और माप कभी भी एक ही संदर्भ विंडो साझा नहीं करते हैं; छिपे हुए ओरेकल का मूल्यांकन एक अलग मॉडल परिवार द्वारा किया जाता है, जिसमें एजेंट की तर्क क्षमता छिपी होती है। मॉडल वह गेम नहीं खेल सकता जिसे वह समझ नहीं सकता।
- **संगति द्वारा विश्वसनीयता।** `पास^के` (सभी *के* स्वतंत्र परीक्षण सफल), विल्सन अंतराल, और क्रॉस-फैमिली जज पैनल - वितरण की रिपोर्ट करने के लिए बनाए गए, न कि बिंदु अनुमान।
- **एक गतिशील सूची।** हल की गई पहेलियों को `रिग्रेशन` में *घटाया* जाता है, कभी भी हटाया नहीं जाता है, इसलिए सूची एक क्षमता-विकास समयरेखा बन जाती है क्योंकि सीमा आगे बढ़ती है।

## खतरा मॉडल (सारांश)

क्रूसिबल का प्राथमिक विरोधी **माप के तहत मॉडल** है, जो उत्तर कुंजी तक पहुंचने या मूल्यांकन का पता लगाने और उसमें हेरफेर करने का प्रयास कर सकता है। यह **निम्नलिखित तक पहुंचता है**: पहेली प्रॉम्प्ट/सेटअप, सॉल्वर की अपनी सीमित कार्य निर्देशिका, और मॉडल-प्रदाता एपीआई कॉल जो ऑपरेटर कॉन्फ़िगर करता है। यह **निम्नलिखित तक नहीं पहुंचता है**: ओरेकल/उत्तर कुंजी (एक अलग होस्ट पर अलग से मूल्यांकित किया जाता है जिस तक सॉल्वर नहीं पहुंच सकता) या प्रेरणा "क्रोम" (रैंक/स्थिति - कभी भी स्कोर किए गए संदर्भ में इंजेक्ट नहीं किया जाता)। **अनुमतियाँ**: रनटाइम पर पर्यावरण चर के माध्यम से मॉडल-प्रदाता कुंजियाँ; कोई बंडल रहस्य नहीं, कोई टेलीमेट्री नहीं, कोई स्वयं की आउटबाउंड कॉल नहीं। पूर्ण प्रकटीकरण - जिसमें यह भी शामिल है कि एक सीमा *गहन रक्षा* है, न कि एक कठोर गारंटी - **[SECURITY.md](SECURITY.md)** में है।

## आर्किटेक्चर

क्रूसिबल **[इंस्पेक्ट एआई](https://inspect.aisi.org.uk/)** (यूके एआईएसआई) पर एक **पतली नीति परत** है, न कि स्क्रैच से बनाया गया एक ढांचा। एक एकल `अटेंप्टस्टेट` ऑब्जेक्ट को डिजाइनर → सॉल्वर → (आलोचक) → जज के माध्यम से **एक `जेनरेट` चोक पॉइंट** से जोड़ा जाता है, ताकि प्रत्येक मॉडल और उपकरण कॉल देखने योग्य हो।

| मॉड्यूल | जिम्मेदारी |
| ------ | -------------- |
| `puzzle_loader` | एक पहेली निर्देशिका (`meta.json` / `prompt` / `setup_script`) को सॉल्वर-दृश्यमान स्थिति में लोड करता है। **कभी भी ओरेकल को नहीं छूता।** |
| `sandbox` | एक लॉक किए गए, नेटवर्क-रहित कंटेनर में `exec` / `read_file` / `write_file` चैनल को संकुचित करें। |
| `roles` | पांच भूमिका स्लॉट (डिजाइनर / सॉल्वर / आलोचक / जज / कोहॉर्टसॉल्वर)। केवल सॉल्वर को उपकरण मिलते हैं; आलोचक इंटरफ़ेस-आरक्षित है, डिफ़ॉल्ट रूप से बंद है। |
| `budget_governor` | प्रति-वर्ग उपकरण-कॉल + दीवार-घड़ी बजट, एजेंट को प्रदर्शित किया जाता है, कर्नेल-साइड पर लागू किया जाता है; रोग संबंधी लूप पर कठोर समाप्ति। |
| `oracle_scorer` | आउट-ऑफ-बैंड ग्रेडिंग: छिपे हुए ओरेकल के विरुद्ध हल किया गया **और** कोई रिग्रेशन नहीं (SWE-बेंच पैटर्न)। |
| `judge_panel` | नवीनता सत्यापन और बाईपास का पता लगाने के लिए मॉडल-स्कोरर + रिड्यूसर के क्रॉस-फैमिली पैनल (PoLL)। |
| `trace_writer` | प्रति-प्रयास प्रतिलिपि, इंस्पेक्ट `EvalLog` आकार में; बड़े ब्लॉब्स को डाइजेस्ट द्वारा संग्रहीत किया जाता है। |
| `observability` | प्रति-प्रयास → प्रति-पहेली → प्रति-मॉडल रोलअप; `पास^के` मूल। |
| `attestation` | क्रिप्टोग्राफ़िक उत्पत्ति (कोसाइन + इवेंट-स्टोर) एक टाइप किए गए सबप्रोसेस सीमा के पीछे। |

सीलबंद सीमा तीन स्तरों में चलती है - **स्तर 1** स्कोर किया गया संदर्भ (तैनाती-आकार, फ्रेमिंग-तटस्थ), **स्तर 2** जुड़ाव फ्रेमिंग (प्रत्येक रिलीज के लिए संदूषण के लिए जांच की जाती है), **स्तर 3** क्रोम (रैंक/लीडरबोर्ड - केवल मानव-सामना करने वाला यूआई, कभी भी उस संदर्भ में नहीं जहां मॉडल हल करता है)। पूर्ण डिजाइन तर्क, उद्धरणों के साथ, [`docs/research-grounding.md`](docs/research-grounding.md) में है।

## त्वरित शुरुआत

क्रूसिबल पर्यावरण और निर्भरता प्रबंधन के लिए [`uv`](https://docs.astral.sh/uv/) का उपयोग करता है। पायथन **3.11+**।

```bash
# Create the venv and install the dev + stats extras
uv sync --extra dev --extra stats

# Run the test suite (with the coverage gate)
uv run pytest --cov=ai-crucible --cov-report=term-missing

# Lint
uv run ruff check .

# One command: lint + tests + build + smoke
bash verify.sh
```
